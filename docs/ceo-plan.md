## Vision

### 10x Check（将来フェーズのビジョン — 現行スコープ外）
今のプランが「WHY を記録するプロキシ」だとすると、10x は **AI アクション登録簿（AI Action Registry）** — 改ざん不可能で法的に提出可能な監査証跡インフラ。エージェントが API を叩くたびに、LLM の推論チェーンが構造化される：意図カテゴリ（データ参照・変更・削除・外部通信）、触れたデータ分類（個人情報・財務・医療）、リスクスコア、暗号署名。ダッシュボードはコンプライアンス管制センターになる：リアルタイム異常検知・ポリシーゲート（WHY なしの tool_use 呼び出しをブロック）・個人情報保護法/金融庁ガイドライン準拠の自動レポート生成。Microsoft の「防御・封鎖」vs Agent Data Gateway の「説明可能性の標準」という差別化軸が際立つ。

## WHO / WHAT / WHY の定義
- **WHO**: リクエストを送信したエージェントの識別子。優先順位: (1) リクエストボディの `metadata.user_id` フィールド、(2) `X-Agent-ID` リクエストヘッダ、(3) なければ "unknown"。API キーのハッシュは使用しない（Anthropic キーは全て `sk-ant-api03-` で始まるため先頭バイトのエントロピーがゼロに近く識別不能）。
- **WHAT**: `tool_use` ブロックの `name` フィールド（例: `get_customer`, `delete_record`）。
- **WHY**: `tool_use` ブロックの直前に存在する `text` ブロックの内容（Plan 1）。存在しない場合は null（ダッシュボードでフラグ表示）。

## WHY 抽出の仕組み（Plan 1 / Plan 2）
- **Plan 1（優先）**: HTTP プロキシが Anthropic API レスポンスを受信後、`content` 配列を走査し、`tool_use` ブロック直前の `text` ブロックを WHY として抽出。エージェントへの変更不要。
- **Plan 2（フォールバック）**: LLM のシステムプロンプトに「ツール呼び出し前に必ず一行で理由を述べよ」を追加し、その発言を WHY として記録。エージェントへのシステムプロンプト変更が必要。
- **切り替え判断期限**: 5月末（ハッカソン2週間前）。Plan 1 が安定しなければ Plan 2 へ切り替える。詳細は設計書 Technical Risk & Fallback セクション参照。

## MVP 完了基準（ハッカソンデモ）
以下のシナリオを Docker Compose 1コマンドで起動した環境で通過できること：
1. Anthropic SDK のエンドポイントを localhost:8080 に変更したエージェントがツール呼び出しを実行できる
2. ダッシュボード（localhost:3000）に WHO / WHAT / WHY の一覧が日本語で表示される
3. WHY が存在する行と欠落している行の両方がデモ中に確認できる

## デモシナリオ（D8+D9 改訂）
**プロジェクト管理エージェント**: 「バグ報告 #456 を調査して、影響範囲を確認し、対応チケットを作成して担当者にアサインして」という依頼を受けるエージェント。書き込み・削除系操作を含めることで「なぜそのAPIを呼んだか」の説明価値が際立つ。
- 使用ツール（読み取り）: `search_issues` / `get_issue` / `get_commit`
- 使用ツール（書き込み）: `create_issue` / `assign_issue` / `add_comment`（モック API）
- WHY の自然発生例: 「バグの影響範囲を特定するために」「重要度が高いと判断したため優先度Highでチケットを作成するために」「担当者が空き状況を確認するために」
- WHO: `X-Agent-ID: project-mgmt-agent` ヘッダで識別
- デモの核心: 「なぜ create_issue を呼んだか」がダッシュボードに表示される体験 → CTO/CISO が「これがあれば説明できる」と言う体験
- 実装: 読み取りは実 GitHub API（Personal Access Token）、書き込みはモック API エンドポイント（実害なし）

## Accepted Scope（このプランに追加）
- **SSE ストリーミング中継対応**: `stream: true` リクエストを正しく中継し、WHY 抽出は `event: message_stop` 受信後にバッファ全体を再構成（SSE イベント列 → `content[]` 配列）してから走査。非ストリーミングリクエスト（`stream: false` または省略）は同期で全レスポンスを受信・走査するため実装は単純。バッファ上限: 16MB または 60 秒のタイムアウトのいずれか先に達した時点でバッファを破棄し WHY = null として記録、以降のチャンクはバッファリングせず直接クライアントへ中継する（上限値は環境変数で変更可）。
- **WHY 欠落自動フラグ**: WHY が null または Unicode コードポイント数 20 未満の行をダッシュボードでハイライト（20 は暫定値、環境変数で変更可）。Plan 1/Plan 2 の切り替え判断基準が目視で確認できる。
- **永続化 / スキーマ**: SQLite（Docker Compose の volume mount でコンテナ再起動後もデータ保持）。最低限のカラム: `id, timestamp, who, what, why, flag_reason, streaming`。
- **WHO の保存**: metadata.user_id → X-Agent-ID → "unknown" の優先順位で取得した識別子をそのまま保存。API キーは保存しない。
- **プロキシプロトコル**: ローカル開発環境で証明書管理を不要にするため平文 HTTP で Listen（`ANTHROPIC_BASE_URL=http://localhost:8080`）。Anthropic API への上流通信は HTTPS を維持する。
- **認証ヘッダ**: クライアントの `Authorization` / `x-api-key` / `anthropic-version` / `anthropic-beta` ヘッダを上流へ pass-through する。プロキシ自身は API キーを保持しない。（`anthropic-beta` は拡張思考・プロンプトキャッシュ等のベータ機能有効化に使われるため必須）
- **上流エラー処理**: Anthropic API からの 4xx/5xx/タイムアウトはそのままクライアントへ中継し、エラーレスポンスも SQLite に記録する（WHY = null、flag_reason = "upstream_error"）。
- **ダッシュボード UI**: Hono + 静的 HTML（TypeScript/Bun スタック統一）。port 3000 で Listen。SQLite を read-only 参照。
- **複数 tool_use の WHY 紐付け**: 各 `tool_use` に対し、同一レスポンスの `content[]` 内で最も近い直前の `text` ブロックを WHY とする。複数 `tool_use` が連続する場合、直前の `text` が存在する最初の `tool_use` のみが WHY を持ち、残りは WHY = null となる。

## Eng Review Additions (2026-05-04, /plan-eng-review)

### Architecture
- **A1 (Schema)**: SQLite 行に `text_context TEXT`（直前 text ブロック全文）と `raw_response BLOB`（生 JSON）を追加。DESIGN.md の WHY クリック展開 UI を実装可能にする。`extraction_source ENUM('text','thinking','fallback','none')` も追加して WHY ソースを記録
- **A2 (SSE tee)**: 上流 SSE ストリームを `ReadableStream.tee()` で 2 本に分割。(1) クライアントへ即時中継、(2) WHY 抽出バッファ。バッファ後遅延中継を防ぐ
- **A3 (Failure modes)**:
  - 上流タイムアウト: 60s default（env `UPSTREAM_TIMEOUT_MS` で変更可）。504 を返却、`flag_reason="upstream_timeout"`
  - SQLite 書き込み失敗（disk full 等）: レスポンス中継は続行、stderr エラーログ、post-mortem キュー（オプション、in-memory）
  - クライアント切断 mid-SSE: `AbortController` で上流 fetch を中止、`flag_reason="client_disconnect"` で記録
- **A4 (Mock API)**: `mock-api/` Hono サービスとして同梱。`POST /issues`, `POST /issues/:id/assign`, `POST /issues/:id/comment`。in-memory state、docker-compose に 1 service 追加

### Code Organization
- **A5 (Monorepo workspaces)**: Bun workspaces で `packages/shared/` に SQLite 行型・Anthropic API リクエスト/レスポンス型を集約。`proxy/`, `dashboard/`, `mock-api/` から共通 import

### Tests (CRITICAL — Boil the Lake)
- **A6 (Test scope)**: テストレビューで生成したカバレッジダイアグラム上の **30+ パスすべてを必須**。実装と同時にテストを書く方針。テスト計画は `~/.gstack/projects/covelline-agent-data-gateway/numa08-main-eng-review-test-plan-20260504-223028.md`

### Performance / Resilience
- **A7 (Memory bound)**: env `MAX_CONCURRENT_BUFFERS=32`（default）。超過時の追加 SSE リクエストはバッファ無し中継のみ、`WHY=null` + `flag_reason="buffer_capacity"`
- **A10 (tee backpressure)**: tee 実装は **drop-on-overflow buffer** ポリシー。クライアント側 backpressure で上流をブロックしない。バッファ上限超過時はチャンクを drop し flag。これによりクライアント遅延 → OOM 連鎖を防ぐ

### WHY Extraction
- **A9 (Thinking blocks)**: `extended thinking` 有効時は推論が `text` でなく `thinking` ブロックに入る。WHY 抽出ロジックを「`tool_use` 直前の `text` または `thinking` ブロック」に拡張。`extraction_source` カラムにソース種別を記録

### Storage Lifecycle
- **A13 (BLOB retention)**: env `RAW_RESPONSE_TTL_DAYS=90`（default）。Nightly scheduled task で TTL 超過行の `raw_response` を NULL クリア（`text_context` と `why` は保持）。SQLite サイズの暴走を防ぐ。コンプライアンス完全保持期間（7年）が必要な顧客は env で延長
